### Задание 1. Кеширование — примеры проблем, которые решает кеширование
```
Высокая нагрузка на базу данных (БД):

	Проблема: При большом количестве пользователей частые запросы к БД (например, за популярной новостью или товаром) создают пиковую нагрузку, замедляя работу системы.

	Решение: Кеш хранит «горячие» данные. Вместо 10 000 запросов в БД — один запрос в БД для обновления кеша и 9 999 чтений из оперативной памяти.

Высокая задержка ответа (High Latency):

	Проблема: Чтение с диска (HDD/SSD) или выполнение сложных JOIN-запросов в SQL занимает десятки и сотни миллисекунд.

	Решение: Кеш (RAM) отдаёт данные за 0.5–2 мс, что критически важно для real-time приложений (API, веб-сайты).

«Тяжёлые» вычисления:

	Проблема: Обработка изображений, рейтингов, ML-предсказаний или агрегация данных (SUM, COUNT) тратит CPU.

	Решение: Результат вычислений сохраняется в кеш. При следующем таком же запросе данные просто отдаются из памяти.

Thundering Herd (Эффект «Слона на распродаже»):

	Проблема: При падении кеша (или окончании TTL) тысячи клиентов одновременно бросаются пересчитывать/грузить данные в БД, убивая её.

	Решение: «Блокирующий» кеш (например, SETNX в Redis) или «ранний пересчёт» (stale-while-revalidate).

Hot Key (Один супер-популярный ключ):

	Проблема: Профиль звёзды YouTube/Instagram запрашивается миллионами пользователей в секунду.

	Решение: Репликация ключа на несколько кеш-нод или локальный кеш (in-memory cache в приложении).
```
